## Critical Issues

### 1. Parent-child chunking does not exist — must be implemented

**What the audit found:**  
The current `ParentChildChunker` (src/chunking.py) creates exactly 1 chunk per company row. The `parent_id` on each chunk is the `record_id` of that same chunk (a 1:1 mapping). No child decomposition occurs. The label "parent-child" is architecturally incorrect.

**Impact:** Without true parent-child chunking, experiments 4–6 cannot be run and the six-experiment comparison is impossible.

**Fix required:**  
Add `TrueParentChildChunker` to `src/chunking.py`. Each company record produces 3 child chunks:
- `identity`: Company + Location + Tier Level + Status
- `industry`: Industry + Product/Service  
- `profile`: Employment + OEM Status + EV Relevant + Primary OEMs

Only child chunks are indexed in ChromaDB and BM25. On retrieval, child chunks map back to the full parent company record via `parent_lookup`. The LLM always receives the full parent text.

ChromaDB collection for parent-child: `gnem_child_chunks` (separate from existing `gnem_chunks`).

**Status:** Not fixed yet — planned in `scripts/run_experiment.py` + `src/chunking.py` changes.

## Minor Issues

### 4. Vector similarity inversion formula is mathematically incorrect for cosine distance

**What the audit found:**  
`normalize_vector_scores()` in `src/retrieval.py` uses `1/(1+distance)`. ChromaDB returns cosine distance ∈ [0, 2]. The correct formula for cosine distance is `1 - distance/2`. The subsequent min-max normalization partially mitigates this by rescaling all scores, so relative ranking is preserved. Absolute similarity scores are not true cosine similarities.

**Impact on research:** Low for relative comparisons (dense vs. sparse vs. hybrid). Medium for absolute score interpretation or cross-system comparison.

**Fix:** Change to `1 - distance/2` in `src/retrieval.py`.

**Timing constraint:** Do NOT fix before all six experiments are run. If fixed mid-experiment, normal_* and parent_child_* runs will use different score normalization, making comparison invalid. Fix only after all six experiments are complete, then note the discrepancy.

### 5. BM25 index rebuilt on every run

**Impact:** Performance only — startup takes longer. Results are identical each run (deterministic).  
**Fix:** Persist BM25 index with pickle. Low priority.

### 6. Context_Sparse has 3949 rows instead of 4000

**Cause:** BM25 returned fewer than 80 results for some queries (score = 0.0 excluded).  
**Impact:** Some sparse questions have fewer than 80 retrieved contexts. May slightly reduce sparse performance metrics on those questions.  
**Fix:** Investigate which questions have < 80 sparse results. Document in fairness check.

### 7. questions.xlsx lacks question_id column

**Impact:** question_ids are auto-generated by row order (Q001–Q050). Fragile if rows are ever reordered.  
**Fix:** Add explicit `question_id` column to `data/questions.xlsx`, or consistently use `data/questions.csv` which already has it.

### 8. ragas missing from requirements.txt

**Status:** Fixed in previous session — `ragas>=0.4.3` added to requirements.txt.
