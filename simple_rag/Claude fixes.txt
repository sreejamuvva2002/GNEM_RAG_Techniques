  ---
  Problems or Risks Found

  [CRITICAL] RAGAS score merging uses positional assignment. In
  evaluate_all_experiments.py, _run_ragas_for_experiment():

  meta_df[target_col] = scores_df[ragas_col].values

  scores_df is the DataFrame returned by ragas.evaluate(). Even with
  raise_exceptions=False, RAGAS may drop rows for which it cannot compute a
  metric (LLM timeout, parse failure, etc.). If it drops any row, scores_df has
  fewer rows than meta_df. The positional .values assignment then shifts every
  score from the first dropped row onward — Q020 gets Q019's score, Q021 gets
  Q020's score, and so on — silently, with no error raised. This is a real bug
  that would make per-question scores and ranking analysis incorrect.

  [MEDIUM] Vector normalization formula is wrong for cosine distance.
  src/retrieval.py line 84:

  similarities = 1.0 / (1.0 + distances.astype(np.float64))

  ChromaDB returns cosine distances in [0, 2]. The correct similarity transform
  for cosine distance is 1 - dist/2, yielding [0.0, 1.0]. The formula 1/(1+dist)
   produces [0.33, 1.0] — compressed and non-linear. This does not corrupt
  relative ranking, but it means hybrid score fusion weights are applied to a
  non-standard scale. The subsequent min-max normalization partially
  compensates, but absolute similarity scores are not meaningful. This is
  flagged — do not fix it until all 6 runs complete, since fixing it
  mid-experiment would make scores inconsistent across runs.

  [MEDIUM] embed_query called unconditionally in sparse mode. src/retrieval.py
  calls the embedding model even when mode="sparse", wasting approximately 100ms
   per question per experiment run. This has no correctness impact but inflates
  runtime for sparse experiments unnecessarily.

  [LOW] Sparse context gap. BM25's score > 0.0 filter drops terms not present in
   any document. Q019 returns only 49 contexts (not 80); Q029 returns 60. These
  questions likely use rare or absent vocabulary. This is a real behavioral
  difference between sparse and dense retrieval — not a bug, but a property to
  note when interpreting retrieval metrics. The normal_sparse experiment already
   exhibits this.

  [LOW] SYSTEM_PROMPT contains unfilled placeholders. src/llm.py defines
  SYSTEM_PROMPT with {question} and {context} in the string, but these are never
   interpolated — the system prompt is sent as a static string with literal
  {question} and {context} text. The actual question and context are correctly
  placed in the user message. This is cosmetic only and does not affect
  generation, but it is misleading when reading the code.

  [LOW] Golden answer alignment is fragile. The row-order mapping works
  correctly for the current file. But if anyone filters, sorts, or reorders
  Human validated 50 questions.xlsx before evaluation, the alignment will
  silently break — Q001 would get the wrong golden answer. There is no
  question-text fuzzy match as a safety check.

  Research Validity Assessment

  The 6-experiment design is internally valid for comparing retrieval strategies
   on this dataset:

  - The same 50 questions and golden answers are used for all 6 experiments.
  - The LLM, prompt, and temperature are identical — any answer quality
  difference is attributable to retrieval strategy, not generation.
  - Normal and parent-child chunking differ in what gets indexed and how context
   is assembled, which is the intended comparison axis.
  - ChromaDB collection separation guarantees there is no cross-contamination
  between chunking strategies.

  Absolute metric values should be interpreted cautiously because:

  - RAGAS faithfulness and answer_relevancy themselves depend on an LLM judge
  (gemma3:27b or whatever is configured via RAGAS_LLM_MODEL). Results may vary
  across model versions.
  - The vector normalization formula affects hybrid score magnitude, not
  ranking. Relative comparisons between experiments are valid; absolute
  similarity scores are not.
  - The sparse context gap (BM25 returning <80 contexts for some questions)
  means sparse and hybrid experiments retrieve a smaller effective context pool
  for those questions. This is a property of the retrieval mode, not a flaw.

  Claims that can be supported by these results:

  - Comparison of dense vs. sparse vs. hybrid retrieval under identical
  generation conditions.
  - Comparison of row-level chunking vs. field-group child chunking under
  identical retrieval and generation conditions.
  - Ranking of experiments by RAGAS metrics (relative, not absolute).

  Claims that cannot be supported:

  - Absolute correctness rates without fixing the RAGAS score alignment bug
  first.
  - Sub-question-level analysis if RAGAS drops any rows (alignment bug).
